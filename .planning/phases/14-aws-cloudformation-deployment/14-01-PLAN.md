Create `Dockerfile` at project root with two stages:

**Stage 1: builder**
```dockerfile
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY tsconfig.server.json ./
COPY server/ ./server/
COPY db/ ./db/
RUN npx tsc -p tsconfig.server.json --noEmit false --outDir dist-server
```

**Stage 2: production**
```dockerfile
FROM node:22-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=builder /app/dist-server ./dist-server
COPY --from=builder /app/db ./db
COPY server/controllers/api/data/datasets/ ./server/controllers/api/data/datasets/
ENV NODE_ENV=production
ENV PORT=2700
EXPOSE 2700
CMD ["node", "dist-server/server.js"]
```

Key decisions in the Dockerfile:
- `node:22-alpine` for small image size (~180MB vs ~1GB for full node)
- `npm ci` in builder (all deps including devDeps for tsc)
- `npm ci --omit=dev` in production (only runtime deps)
- `tsc -p tsconfig.server.json --noEmit false --outDir dist-server` overrides the noEmit:true in tsconfig without modifying the file
- Copy `db/` directory for knex migrations at runtime (db:migrate may run in container)
- Copy `server/controllers/api/data/datasets/` for JSON data files loaded by dataController.ts at runtime
- Do NOT copy `src/` (frontend code — not needed in API container)
- Do NOT use pm2 or tsx — Docker is the process supervisor, node runs compiled JS
- Do NOT add HEALTHCHECK — ECS uses its own health check configuration

<task type="auto">
  <name>Task 3: Add trust proxy setting for CloudFront</name>
  <files>server/server.ts</files>
  <read_first>
    - server/server.ts (find where Express app is created and middleware is applied)
  </read_first>
  <action>
Add `app.set('trust proxy', 1)` to server.ts immediately after the Express app is created (`const app: Express = express();`).

This is REQUIRED because CloudFront sits in front of the Express server. Without it:
- `express-rate-limit` sees all requests as coming from CloudFront's edge IP (one IP) and rate-limits all users as a single client
- `req.ip` returns CloudFront's IP instead of the real visitor's IP
- Phase 12's per-IP rate limiting (carefully tuned) breaks completely

With `trust proxy` set to 1, Express reads the `X-Forwarded-For` header that CloudFront sets, and `req.ip` returns the real visitor's IP.

The value `1` means "trust the first proxy hop" — which is CloudFront. This is the correct setting for a single-proxy architecture (CloudFront → Fargate, no ALB in between).

Add after `const app: Express = express();`:
```typescript
// Trust CloudFront proxy — required for correct req.ip and rate limiting
app.set('trust proxy', 1);
```

2. Modify `server/ingestion/retryRunner.ts` to invalidate CloudFront cache after a successful ingestion.

Add at the top:
```typescript
import { CloudFrontClient, CreateInvalidationCommand } from '@aws-sdk/client-cloudfront';
```

Add a helper function after imports:
```typescript
async function invalidateCloudFrontCache(): Promise<void> {
  const distId = process.env.CF_DIST_ID;
  if (!distId) {
    console.log('[cache] CF_DIST_ID not set — skipping CloudFront invalidation (local dev)');
    return;
  }

  try {
    const cf = new CloudFrontClient({ region: 'us-east-1' });
    await cf.send(new CreateInvalidationCommand({
      DistributionId: distId,
      InvalidationBatch: {
        CallerReference: `ingestion-${Date.now()}`,
        Paths: { Quantity: 1, Items: ['/data/*'] }
      }
    }));
    console.log('[cache] CloudFront /data/* cache invalidated');
  } catch (err) {
    // Log but don't fail the ingestion — data is in the DB, cache will expire naturally
    console.error('[cache] CloudFront invalidation failed:', (err as Error).message);
  }
}
```

3. Call `invalidateCloudFrontCache()` after a successful ingestion in `runWithRetry`. Add after `return result;` on the success path (line 24):

```typescript
      const result = await fn();
      if (attempt > 1) {
        console.log(`[${source}] Succeeded on attempt ${attempt}/${MAX_RETRIES}`);
      }
      // Bust CloudFront cache so users see fresh data
      await invalidateCloudFrontCache();
      return result;
```

Key design decisions:
- **Single integration point**: All 7 cron jobs (UNHCR, IOM, Eurostat, Frontex, CBP, UK Channel, ACLED) go through `runWithRetry` — adding invalidation here covers all of them automatically
- **Graceful failure**: If the CloudFront invalidation fails (network issue, wrong credentials), the ingestion still succeeds. Data is in the DB — the cache will expire naturally at the 1-hour TTL
- **No-op in local dev**: If `CF_DIST_ID` is not set (local development), the function logs a skip message and returns. No AWS SDK calls, no errors
- **Invalidation path**: `/data/*` matches the CloudFront cache behavior path pattern, clearing all cached API responses
- **Cost**: $0 for the first 1,000 invalidation paths/month (7 cron jobs × 4 weeks = ~28 invalidations/month — well within free tier)
